TEST02:

FORMAT VDISK WITH INIT LLFS

First section of memory is the super block.
This is formatted according to the assignment writeup
(i.e. Three ints: magic number, number of blocks on disk, number of inodes on disk)

Second section of memory is the free block.
This is formatted according to the assignment writeup
(i.e. A set bit indicates that the block is free whereas a zero bit
indicates it is reserved, notice that there is a d in row 2 indicating that the root's data block is reserved)

Address 0x1400 (Block index 10):
This is an inode for the root directory.
The first 4 bytes indicate the file size (which
in this case is 0x20 = 32 = ENTRY_SIZE because it has one entry).
The next int (4 bytes) is the flag, which is set to 1, which identifies a directory.
Every two bytes after this is a block number.
In this case the first block number is 0x8a, which, when multiplied by BLOCKSIZE, gives the next memory address displayed.

Address 0x11400 (Block index 138):
This is a data block for the root directory.
This is formatted according to the assignment writeup.
That is, the first byte indicates the inode id (in this case 1, for the root)
and the next 31 bytes are the file name of the entry (which is simply '.' in this case).

TEST03:

OPEN A FILE IN ROOT DIRECTORY (CREATE)

Filename: /hello
File descriptor (inode id): 2
Filesize Before: 0 bytes

WRITE '$' SYMBOL TO FILE (1.5MB)

Filesize After: 1500000 bytes

The wacky hexdump actually means it's working
2929.6875 blocks should be used for data,
11.40625 blocks for single indirection,
1 double indirection block,
and 1 file inode (total = 2944 blocks)

Therefore the free block should display that the first 12 blocks are taken,
because the first 10 are reserved + the root inode + our file's inode,
and 2944 data blocks are reserved from the data block area
(1 block for root directory and 2943 for our file's data blocks).
This can be verified by looking at the free block and calculating that there are
exactly 2944 bits of zero value in the data block area.

Notice that the root inode now has a filesize of 0x40 (two entries).
The inode for the "hello" file has it's filesize in the first int as well (which is much larger)
and it has direct block adresses as well as indirect (single & double) block addresses.
 The non dollar sign sections of memory (after the inodes) are the single and double indirect data blocks.
Feel free to inspect the addresses (although you probably don't want to at this point), they all line up quite beautifully.
 You can also count that there are 12 single indirect blocks and one double indirect block.
